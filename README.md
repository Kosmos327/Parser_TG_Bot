# Parser_TG_Bot

MVP v1 Telegram lead parser bot: Telethon user session читает доступные Telegram-чаты/каналы, keyword-фильтр ищет потенциальные заявки, aiogram-бот уведомляет администратора и даёт команды управления парсером.

Базовый сценарий: аккаунт видит сообщение с фразой вроде `как понизить налоги`, бот сохраняет заявку локально и отправляет уведомление админу.

## Возможности MVP v1

- Чтение новых сообщений через Telethon user session.
- Поиск ключевых фраз без учёта регистра, с нормализацией пробелов и буквы `ё`.
- Dry-run режим для безопасной проверки без уведомлений админу.
- Расширенные фильтры по стоп-фразам, длине сообщения, ботам, forwarded-сообщениям и названиям источников.
- JSON-дедупликация уже обработанных сообщений.
- Локальное JSONL-хранилище найденных заявок.
- Уведомления админу через Telegram Bot API.
- Управление через команды aiogram-бота без остановки процесса.

## Требования

- Python 3.11+
- Telegram API ID и API HASH с <https://my.telegram.org>
- Telegram-бот, созданный через `@BotFather`

### Пояснение настроек

- `API_ID` и `API_HASH` — данные приложения Telegram с <https://my.telegram.org>.
- `SESSION_NAME` — имя локального файла Telethon-сессии без расширения. Например, `parser_account` создаст файл `parser_account.session`.
- `BOT_TOKEN` — токен бота от `@BotFather`.
- `ADMIN_CHAT_ID` — ID чата, куда отправлять уведомления о найденных заявках.
- `ADMIN_IDS` — список Telegram user id через запятую, кому разрешено управлять ботом командами. Если список пустой, команды разрешены только в чате `ADMIN_CHAT_ID`.
- `SOURCE_CHATS` — список usernames, ссылок или ID через запятую. Если оставить пустым, приложение будет слушать все доступные входящие сообщения/каналы, которые Telethon отдаёт через `events.NewMessage`.
- `KEYWORDS` — ключевые фразы через запятую.
- `DEDUP_FILE` — путь к JSON-файлу с обработанными сообщениями.
- `LEADS_FILE` — путь к JSONL-файлу найденных заявок. Каждая строка — отдельный `LeadEvent` в JSON.
- `RULES_FILE` — путь к JSON-файлу правил парсинга. По умолчанию `data/parser_rules.json`.
- `PARSER_ENABLED` — начальное состояние обработки лидов при запуске. Поддерживаются значения `true/false`, `1/0`, `yes/no`.
- `DRY_RUN` — безопасный режим проверки: лиды сохраняются в `LEADS_FILE`, но уведомления админу не отправляются. В логах будет запись, что лид найден в dry-run.
- `MIN_MESSAGE_LENGTH` — минимальная длина сообщения после trim. Более короткие сообщения не проверяются по ключевым фразам.
- `IGNORE_BOTS` — если `true`, сообщения от отправителей с `sender.bot == True` пропускаются.
- `IGNORE_FORWARDS` — если `true`, forwarded-сообщения пропускаются.
- `EXCLUDE_KEYWORDS` — стоп-фразы через запятую. Если текст содержит любую из них, лид не создаётся даже при совпадении `KEYWORDS`.
- `INCLUDE_SOURCE_TITLES` — фрагменты названий источников через запятую. Если список заполнен, бот обрабатывает только источники, в названии которых есть один из фрагментов.
- `EXCLUDE_SOURCE_TITLES` — фрагменты названий источников через запятую. Если название источника содержит один из фрагментов, сообщение пропускается.
- `MAX_TEXT_LENGTH` — максимальная длина `lead.text` и текстового блока уведомления. Более длинный текст обрезается с добавлением `…`.
- `SOURCE_SEARCH_LIMIT` — сколько кандидатов искать за один запрос, по умолчанию 100.
- `JOIN_BATCH_LIMIT` — максимальное число подписок за один запуск сценария, по умолчанию 10. Значения больше 20 считаются рискованными и показываются в предупреждениях.
- `JOIN_DELAY_SECONDS` — задержка между подписками, по умолчанию 90 секунд. Значения меньше 60 считаются рискованными и показываются в предупреждениях.
- `EXCLUDE_PRIVATE_CHATS` — если `true`, личные переписки исключаются из `/dialogs` и не парсятся как источники.
- `SOURCE_CANDIDATES_FILE` — JSON-файл найденных кандидатов источников.
- `SOURCE_EXPORT_FILE` — TXT-файл со списком публичных username для выдачи админу.
- `LOG_LEVEL` — уровень логирования: `DEBUG`, `INFO`, `WARNING`, `ERROR`, `CRITICAL` или `NOTSET`. Для диагностики причин пропуска сообщений используйте `DEBUG`.

Текстовые фильтры работают без учёта регистра, схлопывают множественные пробелы и считают `ё` эквивалентной `е`.

## Как получить ADMIN_CHAT_ID

Самый простой способ:

1. Напишите любое сообщение вашему боту.
2. Откройте в браузере:

   ```text
   https://api.telegram.org/bot<BOT_TOKEN>/getUpdates
   ```

3. Найдите поле `chat.id` в ответе JSON и укажите это значение в `ADMIN_CHAT_ID`.

Если уведомления должны приходить в группу, добавьте бота в группу и отправьте туда сообщение. Затем снова вызовите `getUpdates` и возьмите `chat.id` группы.

## Создание Telethon user session

Перед первым запуском парсера создайте user session:

```bash
python create_session.py
```

Скрипт попросит телефон, код Telegram и, если включена двухфакторная защита, пароль. После успешной авторизации будет создан файл вида `parser_account.session`.

Скрипт выводит только безопасную информацию об аккаунте: ID, имя и username. `API_HASH`, `BOT_TOKEN` и session string не печатаются.

## Запуск парсера

```bash
python -m app.main
```

Для быстрого теста можно оставить `SOURCE_CHATS` пустым. Тогда приложение будет слушать все доступные события, которые получает Telethon user session.

Когда в доступном чате появится сообщение с одной из ключевых фраз, заявка сохранится в `LEADS_FILE`, а администратору придёт уведомление через Telegram-бота. Если включён `DRY_RUN=true`, заявка сохранится, но уведомление не отправится.

Пример уведомления администратору в личные сообщения:

```html
🆕 <b>Найден потенциальный клиент</b>

<b>Логин:</b> @client_user
<b>Имя:</b> Иван
<b>ID пользователя:</b> 123456789

<b>Что написал:</b>
Подскажите, как понизить налоги для ООО?

<b>Дата и время:</b> 13.05.2026 17:30

<b>Источник:</b> Бухгалтерия и налоги
<b>Ссылка:</b> https://t.me/c/1234567890/10
```

Если у сообщения нет публичной ссылки, в поле `Ссылка` будет показано `нет публичной ссылки`. Пустые поля автора заменяются на `нет`, а неизвестный источник — на `неизвестно`. Пользовательский текст и остальные поля экранируются для HTML; длинный блок `Что написал` обрезается так, чтобы итоговое уведомление оставалось не длиннее безопасного лимита Telegram.

## Как выбрать каналы/чаты для парсинга

Telethon user session видит только те чаты, группы и каналы, к которым подключён аккаунт, использованный при создании `parser_account.session` или другого файла из `SESSION_NAME`. Перед настройкой источников убедитесь, что этот аккаунт подписан на нужный канал или состоит в нужном чате.

Если `SOURCE_CHATS` пустой, парсер слушает все доступные входящие сообщения, которые получает текущая Telethon-сессия. Это удобно для первого запуска, но для рабочей конфигурации обычно лучше явно указать источники.

Посмотреть доступные источники можно CLI-скриптом:

```bash
python list_dialogs.py --limit 500
```

Скрипт выводит название, тип entity, username, ID и готовое значение для `SOURCE_CHATS`. Username предпочтительнее, потому что он понятнее и стабильнее для публичных каналов/чатов. Если username нет, используйте ID как fallback. Поле `access_hash`, если оно показано в консоли, нужно только Telethon внутри сессии: не вставляйте его в `.env`.

Тот же быстрый просмотр первых 20 источников доступен через админ-команду Telegram-бота:

```text
/dialogs
```

Заполнить несколько источников в `.env` можно через запятую:

```env
SOURCE_CHATS=@channel1,@channel2,-1001234567890
```

После изменения `SOURCE_CHATS` в `.env` перезапустите приложение:

```bash
python -m app.main
```

Если ограничения источников меняются через меню **Правила парсинга** — например, через списки «Разрешённые источники» или «Исключённые источники», — эти правила применяются без перезапуска. Но изменение `SOURCE_CHATS` в `.env` требует перезапуска `app.main`, потому что список источников передаётся Telethon при регистрации обработчика новых сообщений.


## Интерактивный поиск источников через бота

В админском меню `/start` и `/help` есть кнопка **🔎 Найти новые источники**. Тот же сценарий запускает команда:

```text
/find_sources
```

Сценарий работает пошагово:

1. Бот просит ввести слова или фразы для поиска каналов/чатов. Можно отправить несколько строк, например `налоги`, `бухгалтерия`, `ип`. За один раз берётся максимум 10 запросов, каждый до 100 символов.
2. Telethon ищет публичные каналы/чаты по каждому запросу с лимитом `SOURCE_SEARCH_LIMIT`. Найденные кандидаты сохраняются в `SOURCE_CANDIDATES_FILE`.
3. Бот формирует обычный TXT-файл `SOURCE_EXPORT_FILE`, где каждый публичный источник указан с новой строки в формате `@channel`. Источники без username остаются в JSON, но не попадают в TXT для массовой подписки.
4. Администратор получает файл `source_candidates.txt` и может продолжить поиск новыми фразами или перейти к подписке.

Команды для просмотра результатов:

- `/source_candidates` — отправить текущий TXT-файл и краткую статистику найденных кандидатов.
- `/source_values` — получить строку `SOURCE_CHATS=...` из найденных публичных username.

После поиска бот спрашивает, подписаться ли на найденные источники:

- **✅ Подписаться на все найденные** — берутся только username-источники, которые ещё не помечены как `joined`, `skipped`, `manual_required` и не имеют ошибки. За один запуск применяется лимит `JOIN_BATCH_LIMIT`, между попытками выдерживается `JOIN_DELAY_SECONDS`.
- **✍️ Выборочно** — администратор присылает список каналов столбиком: `@channel1`, `https://t.me/channel2`, `t.me/channel3` или `channel4`. Бот нормализует их к `@username`, максимум до 100 строк, а затем запускает такую же безопасную партию подписок.
- **❌ Отмена** — сценарий завершается без подписок.

Подписка запускается в фоновой `asyncio`-задаче, поэтому `/status` и `/health` продолжают отвечать и показывают, идёт ли подписка на источники. После завершения бот отправляет отчёт: сколько источников добавлено, где аккаунт уже был подписан, какие требуют ручного действия/капчи и какие завершились ошибкой. Если источник требует капчу, заявку на вступление, approval или другое ручное действие, бот добавляет его в `manual_required` и отправляет файл `manual_required_sources.txt` с просьбой пройти проверку вручную. Бот не решает капчи и не пытается обходить проверки.

> **Важно про безопасность Telegram.** Telegram не публикует официальные безопасные лимиты на массовое вступление в каналы. Не увеличивайте `JOIN_BATCH_LIMIT` и не уменьшайте `JOIN_DELAY_SECONDS` без необходимости. При `FloodWait` бот останавливает подписку, сообщает админу и не пытается обходить ограничение.

Если `EXCLUDE_PRIVATE_CHATS=true`, личные диалоги Telethon не показываются в `/dialogs`, не попадают в список источников и пропускаются основным обработчиком сообщений. Это защищает переписки человек-человек от парсинга.

## Команды Telegram-бота

Команды доступны администраторам, заданным через `ADMIN_IDS`, либо чату `ADMIN_CHAT_ID`, если `ADMIN_IDS` пустой.

- `/start` — краткое описание и список команд.
- `/help` — список команд.
- `/status` — включён/выключен парсер, время запуска, сколько сообщений обработано, сколько заявок найдено, последняя ошибка.
- `/pause` — отключить обработку новых лидов без остановки процесса. Telethon и aiogram продолжают работать, но keyword-проверка и уведомления не выполняются.
- `/resume` — включить обработку новых лидов.
- `/last` — показать последние 5 сохранённых заявок.
- `/last 10` — показать последние 10 сохранённых заявок, максимум 20.
- `/stats` — показать общее количество сохранённых заявок и счётчики текущего запуска.
- `/keywords` — показать текущие слова-триггеры из актуальных правил парсинга.
- `/rules` — открыть меню управления правилами парсинга.
- `/find_sources` — запустить интерактивный поиск новых источников.
- `/source_candidates` — показать найденных кандидатов и получить TXT-файл.
- `/source_values` — получить строку для `SOURCE_CHATS` из найденных публичных username.
- `/sources` — показать текущие источники из `SOURCE_CHATS` или текст `все доступные события`.
- `/config` — показать безопасную конфигурацию без `API_HASH` и `BOT_TOKEN`: состояние парсера, dry-run, фильтры, источники, файлы и уровень логирования.
- `/health` — показать operational health: Bot polling, статус Telethon client, включён/выключен парсер, dry-run, последнюю ошибку, uptime и счётчики текущего запуска.
- `/dialogs` — показать первые 20 чатов/каналов, доступных текущей Telethon-сессии, с готовыми значениями для `SOURCE_CHATS`.

## Правила парсинга

Правила парсинга можно менять прямо из Telegram без перезапуска процесса. В `/start` и `/help` есть inline-кнопка **⚙️ Правила парсинга**, а команда `/rules` открывает то же меню. После сохранения нового значения обновлённые правила остаются в памяти текущего запуска и сразу применяются к новым сообщениям.

Правила хранятся в JSON-файле из настройки `RULES_FILE`, по умолчанию `data/parser_rules.json`. Если файла ещё нет, приложение создаёт его из текущих `.env`-настроек `KEYWORDS`, `EXCLUDE_KEYWORDS`, `INCLUDE_SOURCE_TITLES`, `EXCLUDE_SOURCE_TITLES`, `MIN_MESSAGE_LENGTH`, `IGNORE_BOTS` и `IGNORE_FORWARDS`. После появления файла он становится приоритетным источником критериев.

В меню **⚙️ Правила парсинга** доступны категории:

- **Слова-триггеры** — фразы, по которым создаётся заявка. Если список пустой, парсер не будет находить заявки до добавления новых триггеров.
- **Стоп-слова** — фразы, при наличии которых заявка не создаётся даже при совпадении слова-триггера.
- **Разрешённые источники** — фрагменты названий чатов/каналов. Если список пустой, разрешены все источники.
- **Исключённые источники** — фрагменты названий чатов/каналов, которые нужно пропускать.
- **Мин. длина сообщения** — минимальная длина текста от 1 до 1000 символов.
- **Игнорировать ботов** — переключатель пропуска сообщений от bot-аккаунтов.
- **Игнорировать пересланные** — переключатель пропуска forwarded-сообщений.

Для списковых категорий есть действия **➕ Добавить**, **➖ Удалить** и **🧹 Очистить список**. Дубликаты проверяются без учёта регистра, но сохранённый текст остаётся в том виде, в котором его отправил администратор.

## Где хранятся найденные заявки

Заявки сохраняются локально в файл из настройки `LEADS_FILE`, по умолчанию `data/leads.jsonl`.

Формат — JSON Lines: одна строка равна одной найденной заявке. Это удобно для просмотра, резервного копирования и последующей обработки без подключения БД.

Папка `data/` не должна попадать в git.

## Как безопасно проверить парсер

1. Создать `.env` из `.env.example` и заполнить Telegram-настройки.
2. Для первого теста включить безопасный режим:

   ```env
   DRY_RUN=true
   LOG_LEVEL=DEBUG
   ```

   В этом режиме совпавшие лиды сохраняются в `LEADS_FILE`, но уведомления в Telegram не отправляются. `DEBUG` поможет увидеть причины пропуска сообщений: `too_short`, `source_filtered`, `exclude_keyword`, `no_keyword` и т.д.

3. Создать session:

   ```bash
   python create_session.py
   ```

4. Запустить main:

   ```bash
   python -m app.main
   ```

5. Написать тестовое сообщение с фразой `как понизить налоги` в чате, доступном аккаунту Telethon.
6. Проверить, что заявка появилась в `LEADS_FILE`, а в логах есть сообщение о найденном лиде в dry-run.
7. Проверить команды `/config`, `/health`, `/last` и `/stats` в Telegram-боте.
8. После проверки переключить `DRY_RUN=false`, перезапустить процесс и убедиться, что уведомления приходят администратору.

## Безопасность

Нельзя публиковать или коммитить:

- `.env`
- `*.session`
- `*.session-journal`
- содержимое папки `data/`

Эти файлы добавлены в `.gitignore`. В коде нет хардкода секретов, а логи не печатают `API_HASH`, `BOT_TOKEN` или session string.

## Критерии готовности

- `python create_session.py` запускается и создаёт Telethon session.
- `python -m app.main` запускает парсер и Telegram-бота управления.
- Сообщение с ключевой фразой в доступном чате приводит к сохранению заявки и уведомлению администратору.
- `/pause` и `/resume` переключают обработку лидов без остановки процесса.
- `/last`, `/stats`, `/config` и `/health` показывают сохранённые заявки, статистику, безопасный конфиг и operational health.
- Секреты не попадают в git.
- Код проходит проверку на синтаксические ошибки и unit-тесты.
