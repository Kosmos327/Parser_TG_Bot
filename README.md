# Parser_TG_Bot

MVP v1 Telegram lead parser bot: Telethon user session читает доступные Telegram-чаты/каналы, keyword-фильтр ищет потенциальные заявки, aiogram-бот уведомляет администратора и даёт команды управления парсером.

Базовый сценарий: аккаунт видит сообщение с фразой вроде `как понизить налоги`, бот сохраняет заявку локально и отправляет уведомление админу.

## Возможности MVP v1

- Чтение новых сообщений через Telethon user session.
- Поиск ключевых фраз без учёта регистра, с нормализацией пробелов и буквы `ё`.
- Dry-run режим для безопасной проверки без уведомлений админу.
- Расширенные фильтры по стоп-фразам, длине сообщения, ботам, forwarded-сообщениям и названиям источников.
- JSON-дедупликация уже обработанных сообщений.
- Локальное JSONL-хранилище найденных заявок.
- Уведомления админу через Telegram Bot API.
- Управление через команды aiogram-бота без остановки процесса.

## Требования

- Python 3.11+
- Telegram API ID и API HASH с <https://my.telegram.org>
- Telegram-бот, созданный через `@BotFather`

## Установка

1. Склонируйте репозиторий и перейдите в папку проекта.

2. Создайте виртуальное окружение.

   Windows PowerShell:

   ```powershell
   python -m venv .venv
   .\.venv\Scripts\Activate.ps1
   ```

   Linux/macOS:

   ```bash
   python3 -m venv .venv
   source .venv/bin/activate
   ```

3. Установите зависимости.

   ```bash
   pip install -r requirements.txt
   ```

   Для запуска тестов дополнительно:

   ```bash
   pip install -r requirements-dev.txt
   ```

## Настройка .env

Скопируйте пример настроек:

```bash
cp .env.example .env
```

На Windows можно выполнить:

```powershell
Copy-Item .env.example .env
```

Заполните `.env`:

```env
API_ID=123456
API_HASH=your_api_hash
SESSION_NAME=parser_account
BOT_TOKEN=123456:bot_token
ADMIN_CHAT_ID=123456789
ADMIN_IDS=
SOURCE_CHATS=
KEYWORDS=как понизить налоги,понизить налоги,снизить налоги,оптимизация налогов
DEDUP_FILE=data/processed_messages.json
LEADS_FILE=data/leads.jsonl
PARSER_ENABLED=true
DRY_RUN=false
MIN_MESSAGE_LENGTH=10
IGNORE_BOTS=true
IGNORE_FORWARDS=false
EXCLUDE_KEYWORDS=
INCLUDE_SOURCE_TITLES=
EXCLUDE_SOURCE_TITLES=
MAX_TEXT_LENGTH=3500
LOG_LEVEL=INFO
```

### Пояснение настроек

- `API_ID` и `API_HASH` — данные приложения Telegram с <https://my.telegram.org>.
- `SESSION_NAME` — имя локального файла Telethon-сессии без расширения. Например, `parser_account` создаст файл `parser_account.session`.
- `BOT_TOKEN` — токен бота от `@BotFather`.
- `ADMIN_CHAT_ID` — ID чата, куда отправлять уведомления о найденных заявках.
- `ADMIN_IDS` — список Telegram user id через запятую, кому разрешено управлять ботом командами. Если список пустой, команды разрешены только в чате `ADMIN_CHAT_ID`.
- `SOURCE_CHATS` — список usernames, ссылок или ID через запятую. Если оставить пустым, приложение будет слушать все доступные входящие сообщения/каналы, которые Telethon отдаёт через `events.NewMessage`.
- `KEYWORDS` — ключевые фразы через запятую.
- `DEDUP_FILE` — путь к JSON-файлу с обработанными сообщениями.
- `LEADS_FILE` — путь к JSONL-файлу найденных заявок. Каждая строка — отдельный `LeadEvent` в JSON.
- `PARSER_ENABLED` — начальное состояние обработки лидов при запуске. Поддерживаются значения `true/false`, `1/0`, `yes/no`.
- `DRY_RUN` — безопасный режим проверки: лиды сохраняются в `LEADS_FILE`, но уведомления админу не отправляются. В логах будет запись, что лид найден в dry-run.
- `MIN_MESSAGE_LENGTH` — минимальная длина сообщения после trim. Более короткие сообщения не проверяются по ключевым фразам.
- `IGNORE_BOTS` — если `true`, сообщения от отправителей с `sender.bot == True` пропускаются.
- `IGNORE_FORWARDS` — если `true`, forwarded-сообщения пропускаются.
- `EXCLUDE_KEYWORDS` — стоп-фразы через запятую. Если текст содержит любую из них, лид не создаётся даже при совпадении `KEYWORDS`.
- `INCLUDE_SOURCE_TITLES` — фрагменты названий источников через запятую. Если список заполнен, бот обрабатывает только источники, в названии которых есть один из фрагментов.
- `EXCLUDE_SOURCE_TITLES` — фрагменты названий источников через запятую. Если название источника содержит один из фрагментов, сообщение пропускается.
- `MAX_TEXT_LENGTH` — максимальная длина `lead.text` и текстового блока уведомления. Более длинный текст обрезается с добавлением `…`.
- `LOG_LEVEL` — уровень логирования: `DEBUG`, `INFO`, `WARNING`, `ERROR`, `CRITICAL` или `NOTSET`. Для диагностики причин пропуска сообщений используйте `DEBUG`.

Текстовые фильтры работают без учёта регистра, схлопывают множественные пробелы и считают `ё` эквивалентной `е`.

## Как получить ADMIN_CHAT_ID

Самый простой способ:

1. Напишите любое сообщение вашему боту.
2. Откройте в браузере:

   ```text
   https://api.telegram.org/bot<BOT_TOKEN>/getUpdates
   ```

3. Найдите поле `chat.id` в ответе JSON и укажите это значение в `ADMIN_CHAT_ID`.

Если уведомления должны приходить в группу, добавьте бота в группу и отправьте туда сообщение. Затем снова вызовите `getUpdates` и возьмите `chat.id` группы.

## Создание Telethon user session

Перед первым запуском парсера создайте user session:

```bash
python create_session.py
```

Скрипт попросит телефон, код Telegram и, если включена двухфакторная защита, пароль. После успешной авторизации будет создан файл вида `parser_account.session`.

Скрипт выводит только безопасную информацию об аккаунте: ID, имя и username. `API_HASH`, `BOT_TOKEN` и session string не печатаются.

## Запуск парсера

```bash
python -m app.main
```

Для быстрого теста можно оставить `SOURCE_CHATS` пустым. Тогда приложение будет слушать все доступные события, которые получает Telethon user session.

Когда в доступном чате появится сообщение с одной из ключевых фраз, заявка сохранится в `LEADS_FILE`, а администратору придёт уведомление через Telegram-бота. Если включён `DRY_RUN=true`, заявка сохранится, но уведомление не отправится.

## Команды Telegram-бота

Команды доступны администраторам, заданным через `ADMIN_IDS`, либо чату `ADMIN_CHAT_ID`, если `ADMIN_IDS` пустой.

- `/start` — краткое описание и список команд.
- `/help` — список команд.
- `/status` — включён/выключен парсер, время запуска, сколько сообщений обработано, сколько заявок найдено, последняя ошибка.
- `/pause` — отключить обработку новых лидов без остановки процесса. Telethon и aiogram продолжают работать, но keyword-проверка и уведомления не выполняются.
- `/resume` — включить обработку новых лидов.
- `/last` — показать последние 5 сохранённых заявок.
- `/last 10` — показать последние 10 сохранённых заявок, максимум 20.
- `/stats` — показать общее количество сохранённых заявок и счётчики текущего запуска.
- `/keywords` — показать текущие ключевые фразы.
- `/sources` — показать текущие источники из `SOURCE_CHATS` или текст `все доступные события`.
- `/config` — показать безопасную конфигурацию без `API_HASH` и `BOT_TOKEN`: состояние парсера, dry-run, фильтры, источники, файлы и уровень логирования.
- `/health` — показать operational health: Bot polling, статус Telethon client, включён/выключен парсер, dry-run, последнюю ошибку, uptime и счётчики текущего запуска.

## Где хранятся найденные заявки

Заявки сохраняются локально в файл из настройки `LEADS_FILE`, по умолчанию `data/leads.jsonl`.

Формат — JSON Lines: одна строка равна одной найденной заявке. Это удобно для просмотра, резервного копирования и последующей обработки без подключения БД.

Папка `data/` не должна попадать в git.

## Как безопасно проверить парсер

1. Создать `.env` из `.env.example` и заполнить Telegram-настройки.
2. Для первого теста включить безопасный режим:

   ```env
   DRY_RUN=true
   LOG_LEVEL=DEBUG
   ```

   В этом режиме совпавшие лиды сохраняются в `LEADS_FILE`, но уведомления в Telegram не отправляются. `DEBUG` поможет увидеть причины пропуска сообщений: `too_short`, `source_filtered`, `exclude_keyword`, `no_keyword` и т.д.

3. Создать session:

   ```bash
   python create_session.py
   ```

4. Запустить main:

   ```bash
   python -m app.main
   ```

5. Написать тестовое сообщение с фразой `как понизить налоги` в чате, доступном аккаунту Telethon.
6. Проверить, что заявка появилась в `LEADS_FILE`, а в логах есть сообщение о найденном лиде в dry-run.
7. Проверить команды `/config`, `/health`, `/last` и `/stats` в Telegram-боте.
8. После проверки переключить `DRY_RUN=false`, перезапустить процесс и убедиться, что уведомления приходят администратору.

## Безопасность

Нельзя публиковать или коммитить:

- `.env`
- `*.session`
- `*.session-journal`
- содержимое папки `data/`

Эти файлы добавлены в `.gitignore`. В коде нет хардкода секретов, а логи не печатают `API_HASH`, `BOT_TOKEN` или session string.

## Критерии готовности

- `python create_session.py` запускается и создаёт Telethon session.
- `python -m app.main` запускает парсер и Telegram-бота управления.
- Сообщение с ключевой фразой в доступном чате приводит к сохранению заявки и уведомлению администратору.
- `/pause` и `/resume` переключают обработку лидов без остановки процесса.
- `/last`, `/stats`, `/config` и `/health` показывают сохранённые заявки, статистику, безопасный конфиг и operational health.
- Секреты не попадают в git.
- Код проходит проверку на синтаксические ошибки и unit-тесты.
